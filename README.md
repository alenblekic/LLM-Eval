# LLM Eval Dashboard

A lightweight evaluation harness for validating structured LLM outputs, with a browser UI instead of a CLI. Built around an Amazon PPC use case: classifying search terms and auditing campaign structure, then checking the model's JSON output for validity, schema correctness, hallucinated claims, and accuracy against expected labels.

This is a demo/portfolio project ‚Äî the Amazon data is synthetic, generated by a randomized test-case pool rather than pulled from a live seller account. The harness itself (validation pipeline, checks, scoring, reporting) is built to run the same way against real search term reports and campaign data.

## What it does

Each test case sends a prompt to `llama-3.3-70b-versatile` (via Groq) and runs the response through a short-circuit validation chain:

1. **JSON validity** ‚Äî does the response parse?
2. **Schema** ‚Äî does it have `classification`, `confidence`, `reasoning`?
3. **Allowed value** ‚Äî is the classification one of the allowed labels?
4. **Hallucination check** ‚Äî a simple, explainable word-diff heuristic that flags brand names, prices/numbers, or capitalized proper nouns in the reasoning that don't appear in the input.
5. **Accuracy** ‚Äî does the classification match the expected label? (Informational only ‚Äî doesn't fail the case.)

A case is **PASS** only if it clears checks 1‚Äì4. Mismatched accuracy alone doesn't fail it ‚Äî that's the distinction between "the model broke its contract" and "the model disagreed with expectations."

Two task types share this pipeline:

- **Search-term classification** ‚Äî `High Intent` / `Low Intent` / `Brand` / `Off-Topic`
- **Campaign structure audit** ‚Äî `Well-Structured` / `Needs Fix` / `Missing Insight` (duplicate keywords across ad groups, overly broad ad groups, missed trend signals, etc.)

## Features

- **Randomized test suites** ‚Äî every run composes a fresh mix of baseline, hallucination-bait, adversarial (prompt injection), ambiguous, robustness (emoji/non-English), and campaign-structure cases, so no two runs look the same.
- **Test a single input** ‚Äî run one ad-hoc input through the same validation pipeline used by the full suite, with a mode toggle for classification vs. structure audits.
- **Downloadable Markdown report** ‚Äî summary stats, full results table, error log, and hallucination log with the exact flagged snippets.
- Dark, glossy UI. No framework, no build step ‚Äî plain HTML/CSS/JS served as static files.

## Stack

- Node.js + Express (no framework, no build step)
- `groq-sdk` for the model call, `dotenv` for config
- Plain HTML/CSS/JS frontend, no bundler

## Running locally

```bash
npm install
cp .env.example .env   # add your GROQ_API_KEY
npm start
```

Open `http://localhost:3000`.

## Deploying to Vercel

Vercel has zero-config Express support: it looks for a file that imports `express` and default-exports the app at one of a few conventional locations (`app.js`, `index.js`, `server.js`, or the same under `src/`). `src/app.js` already matches that pattern (`export default app`), so no `vercel.json` or `/api` wrapper is needed ‚Äî deploying the repo as-is is enough. Static files in `public/**` are served directly by Vercel's CDN (Express's own `app.use(express.static(...))` is a no-op there, but still works for local dev).

Set `GROQ_API_KEY` in the Vercel project's environment variables before deploying.

Note: the report endpoint is stateless by design ‚Äî the browser sends back the last run's results, and the server just formats them into Markdown. This avoids relying on in-memory state, which isn't guaranteed to persist between serverless invocations.
